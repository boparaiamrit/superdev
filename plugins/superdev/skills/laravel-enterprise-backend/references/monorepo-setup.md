# Monorepo Setup (Laravel)

Set up the pnpm workspace + Turborepo monorepo that hosts `apps/web`, `apps/api` (Laravel), and `packages/contracts`. Read first in Phase 3, before scaffolding the Laravel app. The key difference from the Nest.js setup: `apps/api` is a **Composer/PHP project, not a pnpm package**. Turbo does not manage it; the root workspace drives the contract-generation step by shelling out to `php artisan typescript:transform`.

## Goal structure

```
<workspace>/
├── apps/
│   ├── web/                 ← Next.js frontend (design-to-nextjs skill; pnpm workspace package)
│   └── api/                 ← Laravel 13 backend (composer.json; NOT a pnpm package)
│       ├── app/
│       ├── config/
│       ├── database/
│       ├── routes/
│       ├── tests/
│       └── composer.json
├── packages/
│   ├── contracts/           ← TS types GENERATED by `php artisan typescript:transform`
│   ├── tsconfig/            ← Shared tsconfig presets
│   └── eslint-config/       ← Shared ESLint config
├── docker-compose.yml       ← Single-node CockroachDB for local dev parity
├── .gitignore
├── package.json             ← Workspace root (pnpm)
├── pnpm-workspace.yaml
└── turbo.json
```

`apps/api` intentionally sits at the same path as the Nest.js counterpart so every sibling skill's `apps/api` path assumption holds. It is kept out of `pnpm-workspace.yaml` — pnpm has nothing to manage there.

## Step 1 — Initialize root

```bash
mkdir <workspace> && cd <workspace>
git init
pnpm init
```

Edit root `package.json`:

```json
{
  "name": "<workspace>",
  "private": true,
  "version": "0.0.0",
  "scripts": {
    "build": "turbo run build",
    "dev": "turbo run dev",
    "lint": "turbo run lint",
    "typecheck": "turbo run typecheck",
    "test": "turbo run test",
    "contracts": "cd apps/api && php artisan typescript:transform",
    "format": "prettier --write .",
    "clean": "turbo run clean && rm -rf node_modules"
  },
  "devDependencies": {
    "turbo": "^2.0.0",
    "prettier": "^3.2.0",
    "typescript": "^5.4.0"
  },
  "packageManager": "pnpm@9.0.0",
  "engines": {
    "node": ">=20"
  }
}
```

The `"contracts"` script is what Turbo's `contracts` task invokes (see Step 3). PHP must be on `$PATH` in the CI/local environment where this runs.

## Step 2 — Workspace config

`pnpm-workspace.yaml` — **`apps/api` is deliberately omitted**; it has no `package.json` managed by pnpm:

```yaml
packages:
  - "apps/web"
  - "packages/*"
```

`apps/web` is listed explicitly (not `apps/*`) so the wildcard does not accidentally try to parse `apps/api/composer.json` as a Node package.

## Step 3 — Turbo config

`turbo.json`:

```json
{
  "$schema": "https://turbo.build/schema.json",
  "globalDependencies": ["**/.env.*local"],
  "tasks": {
    "contracts": {
      "cache": false
    },
    "build": {
      "dependsOn": ["^build", "contracts"],
      "outputs": [".next/**", "!.next/cache/**", "dist/**"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    },
    "lint": {
      "dependsOn": ["^build"]
    },
    "typecheck": {
      "dependsOn": ["^build"]
    },
    "test": {
      "dependsOn": ["^build"],
      "outputs": ["coverage/**"]
    },
    "clean": {
      "cache": false
    }
  }
}
```

Key points:

- `"contracts"` task has `"cache": false` — PHP artisan generates a file from live source; caching would serve stale output.
- `"build"` lists `"contracts"` in `dependsOn` so `packages/contracts` (and thus `apps/web`) receives freshly generated types before any TypeScript compilation runs.
- Turbo executes the `contracts` task by running the `"contracts"` script defined in the root `package.json`, which shells out to `php artisan typescript:transform`.

## Step 6 — Contracts package (generated, not hand-authored)

This is the most important shared package. In the Laravel stack the source of truth lives in PHP — `spatie/laravel-data` classes. `php artisan typescript:transform` writes the TS output here. The frontend imports types from this package, but **never edits the generated file manually**.

```bash
mkdir -p packages/contracts/src
```

`packages/contracts/package.json`:

```json
{
  "name": "@<scope>/contracts",
  "version": "0.0.0",
  "private": true,
  "type": "module",
  "main": "./src/generated.ts",
  "types": "./src/generated.ts",
  "exports": {
    ".": "./src/generated.ts"
  },
  "devDependencies": {
    "@<scope>/tsconfig": "workspace:*",
    "typescript": "^5.4.0"
  }
}
```

`packages/contracts/tsconfig.json`:

```json
{
  "extends": "@<scope>/tsconfig/base.json",
  "compilerOptions": {
    "noEmit": true,
    "rootDir": "src",
    "module": "ESNext",
    "target": "ES2022",
    "moduleResolution": "Bundler"
  },
  "include": ["src/**/*.ts"]
}
```

The package does **not** have a `build` script — there is nothing to compile. `apps/web` imports directly from `src/generated.ts` (type-only at compile time).

### Where generated.ts comes from

`config/typescript-transformer.php` in `apps/api` sets:

```php
'output_file' => base_path('../../packages/contracts/src/generated.ts'),
```

Running `php artisan typescript:transform` (or `pnpm contracts` at the root) writes the full TS type union file there. See `laravel-data-contracts.md` for the transformer config and a sample output.

### Committing generated.ts

**Commit `packages/contracts/src/generated.ts` to the repository.** This is a deliberate choice:

- `apps/web` builds cleanly on CI without PHP installed in the Node build environment.
- Reviewers see the contract diff alongside the PHP Data-class diff in every pull request.
- The file is regenerated automatically on every local and CI run where PHP is available (via the `contracts` Turbo task).

Do not add `generated.ts` to `.gitignore`. Instead, the git diff for any Data-class change should include the corresponding TS type change — treat them as a single atomic commit.

`packages/contracts/src/generated.ts` (example output from the transformer):

```ts
/* This file is auto-generated by `php artisan typescript:transform`. Do not edit by hand. */

export type Industry = 'Technology' | 'Healthcare' | 'Finance' | 'Logistics' | 'Other';

export type Stage = 'New' | 'Qualified' | 'Proposal Sent' | 'Won' | 'Lost';

export type CompanyCountsData = {
    contacts: number;
    open_leads: number;
    won_deals: number;
};

export type LastActivityNoneData = {
    kind: 'None';
};

export type LastActivityEmailSentData = {
    kind: 'Email Sent';
    at: string;
    subject: string;
    label: string;
};

export type LastActivityData = LastActivityNoneData | LastActivityEmailSentData;

export type CompanyData = {
    id: string;
    name: string;
    domain: string | null;
    industry: Industry;
    counts: CompanyCountsData;
    last_activity: LastActivityData;
    created_at: string;
    updated_at: string;
};
```

Notice: `domain: string | null` — nullable is explicit. No `?.` access needed on the frontend. This is enforced by the no-null Pest test in `view-data-pattern.md`.

## Step 7 — Wire apps/web to contracts

In `apps/web/package.json`:

```json
{
  "dependencies": {
    "@<scope>/contracts": "workspace:*"
  },
  "devDependencies": {
    "@<scope>/tsconfig": "workspace:*",
    "@<scope>/eslint-config": "workspace:*"
  }
}
```

`apps/api` does **not** appear here — it has no pnpm dependency on the contracts package. Direction is one-way: PHP upstream writes TS, Next.js downstream reads it. The backend cannot import TS.

After editing `apps/web/package.json`:

```bash
pnpm install
```

## Step 8 — Local dev: CockroachDB via Docker Compose

`docker-compose.yml` at the workspace root provides a single-node CockroachDB for local dev parity with the CockroachDB serverless free tier used in production. (Production deploy is covered by the `laravel-bref-deploy` skill.)

```yaml
version: "3.9"

services:
  cockroachdb:
    image: cockroachdb/cockroach:latest-v24.3
    command: start-single-node --insecure
    ports:
      - "26257:26257"   # SQL wire (pgsql driver)
      - "8080:8080"     # Admin UI
    volumes:
      - cockroachdb_data:/cockroach/cockroach-data
    healthcheck:
      test: ["CMD", "cockroach", "node", "status", "--insecure", "--host=localhost:26257"]
      interval: 10s
      timeout: 5s
      retries: 5

volumes:
  cockroachdb_data:
```

Start it:

```bash
docker compose up -d cockroachdb
```

`apps/api/.env` for local dev (sslmode `disable` is acceptable against the insecure single-node container; never use `disable` against CockroachDB Cloud):

```
DB_CONNECTION=pgsql
DB_HOST=127.0.0.1
DB_PORT=26257
DB_DATABASE=defaultdb
DB_USERNAME=root
DB_PASSWORD=
DB_SSLMODE=disable
```

The CockroachDB single-node container is Postgres-wire-compatible. Running `php artisan migrate` against it exercises the exact same UUID/DDL paths as production. No separate Postgres container is needed.

## Step 10 — Verify the pipeline

```bash
# 1. Install JS deps
pnpm install

# 2. Start local CockroachDB
docker compose up -d cockroachdb

# 3. Run Laravel migrations
cd apps/api && php artisan migrate && cd ../..

# 4. Generate TS contracts (also runnable as `pnpm contracts`)
cd apps/api && php artisan typescript:transform && cd ../..

# 5. Confirm generated.ts was written
cat packages/contracts/src/generated.ts | head -5

# 6. Build the whole monorepo (Turbo will run contracts -> build)
pnpm turbo build
```

If `packages/contracts/src/generated.ts` is present and `apps/web` resolves its `@<scope>/contracts` import, the wiring is correct.

## How this differs from the Nest.js monorepo setup

| Aspect | Nest.js | Laravel |
|---|---|---|
| `apps/api` in pnpm workspace | Yes — it's a Node package | **No** — Composer project; excluded from `pnpm-workspace.yaml` |
| Contract source of truth | Hand-authored Zod schemas in `packages/contracts/src/` | **PHP Data classes** in `apps/api/app/Domains/*/Data/`; generated file in `packages/contracts/src/generated.ts` |
| Contract generation | No generation step; authored manually | **`php artisan typescript:transform`** via `pnpm contracts` / Turbo `contracts` task |
| Turbo `contracts` task | Not present | **Added**; `"cache": false`; `build` depends on it |
| Zod runtime validation | Yes — both apps import and parse | **No** — types only at compile time; validation is server-side in laravel-data |
| `packages/contracts` build script | Yes (`tsc`) | **No** — nothing to compile; direct TS import |
| Commit generated file | N/A (hand-authored) | **Yes** — committed so `apps/web` CI builds without PHP |
| Local database | Docker: Postgres 17 + TimescaleDB + Redis | **Docker: CockroachDB single-node** (no Redis) |

## Anti-patterns

- Adding `apps/api` to `pnpm-workspace.yaml`. pnpm will fail trying to parse `composer.json` as a Node manifest.
- Hand-editing `packages/contracts/src/generated.ts`. It will be overwritten the next time `typescript:transform` runs. Put all type changes in the PHP Data classes.
- Adding `generated.ts` to `.gitignore`. This breaks CI builds that run `pnpm turbo build` without a PHP environment (e.g., pure Node build steps).
- Skipping the `contracts` task when adding a new Data class. Run `pnpm contracts` before committing; the new TS types must land in the same commit as the PHP change.
- Running `pnpm turbo build` before `docker compose up -d cockroachdb` and `php artisan migrate`. The `contracts` task needs `apps/api` fully bootstrapped.
- Using `apps/*` in `pnpm-workspace.yaml` instead of listing `apps/web` explicitly. A wildcard will try to scan `apps/api` and may surface confusing errors.
